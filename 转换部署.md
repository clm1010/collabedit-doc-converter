# collabedit-doc-converter 部署手册

> 本服务负责 **DOCX ⇄ DocModel ⇄ HTML ⇄ PDF** 的互转。PDF 导出由 Puppeteer（Chrome for Testing）驱动，**已完全不依赖 LibreOffice / unoserver**。
>
> 适用版本：`puppeteer ^24.42.0`、`Chrome for Testing 147.0.7727.57`、`Node 20`（Docker）/ Node 22（本地开发）、`Debian 12 bookworm-slim`。

## 1. 架构速览

```
┌─────────────────────────────────────────────────────┐
│ Node.js 20 (Express)                                │
│   /import  : DOCX → DocModel                        │
│   /export  : DocModel → DOCX / PDF                  │
│   /health  : 健康检查                                │
├─────────────────────────────────────────────────────┤
│ Puppeteer 24.x (puppeteer-core 内部)                │
│   ↳ Chrome for Testing 147.x (linux64, headless)    │
│   ↳ PUPPETEER_CACHE_DIR=/home/node/.cache/puppeteer │
├─────────────────────────────────────────────────────┤
│ 字体 (fontconfig)                                    │
│   /usr/share/fonts/custom/  ← ./fonts 复制/挂载      │
│   （simsun 宋体、方正字库、wqy-zenhei 等，~650MB）    │
└─────────────────────────────────────────────────────┘
```

- **Chrome 不走 apt**：Debian 的 `chromium` 包在国内镜像下载非常不稳定（>120MB 经常 EOF）。
- **Chrome 不由容器内下载**：pnpm install 时调用 CDN 下载 ~170MB zip 也常被切断。
- **采用的方案**：**宿主机预下载 `chrome-linux64.zip` → COPY 进镜像**，稳定可离线。

镜像大小约 **1.67GB**，其中：字体 `~686MB`、Chrome `~500MB`、node_modules `~129MB`、系统库 `~69MB`、基础镜像 `~293MB`。

## 2. 开发环境

### 2.1 本地（宿主机直接运行，推荐）

优势：调试快、热更新、无需 Docker。

#### 2.1.1 前置要求

- Node.js **20.x 或 22.x**（推荐 22.17+）
- pnpm（通过 corepack 启用）
- 已安装任意 Chrome / Chromium 浏览器（供 Puppeteer 复用，免下载）

```bash
# 启用 pnpm
corepack enable
corepack prepare pnpm@latest --activate

# 切换到项目目录
cd e:/job-project/collabedit-doc-converter
```

#### 2.1.2 安装依赖

> 首次 `pnpm install` 会自动下载 Puppeteer 自带的 Chrome（~170MB）。若希望跳过（使用宿主机 Chrome 即可），设置环境变量。

**Windows PowerShell：**

```powershell
$env:PUPPETEER_SKIP_DOWNLOAD="true"
pnpm install
```

**Linux / macOS / Git Bash：**

```bash
PUPPETEER_SKIP_DOWNLOAD=true pnpm install
```

如果不设置，可临时指向 npmmirror 加速：

```bash
PUPPETEER_DOWNLOAD_BASE_URL=https://cdn.npmmirror.com/binaries/chrome-for-testing pnpm install
```

#### 2.1.3 配置 `.env`

```bash
cp .env.example .env
```

编辑 `.env`，**显式指向宿主机已安装的 Chrome**（避免 Puppeteer 到处找）：

```dotenv
# Windows
CHROMIUM_EXECUTABLE_PATH=C:\Program Files\Google\Chrome\Application\chrome.exe

# macOS
#CHROMIUM_EXECUTABLE_PATH=/Applications/Google Chrome.app/Contents/MacOS/Google Chrome

# Linux
#CHROMIUM_EXECUTABLE_PATH=/usr/bin/google-chrome
```

#### 2.1.4 启动开发服务

```bash
# 热重载开发模式
pnpm dev

# 或编译后以生产方式启动
pnpm build
pnpm start
```

服务默认监听 `http://localhost:3002`。

#### 2.1.5 冒烟测试

```bash
curl http://localhost:3002/health

curl -X POST http://localhost:3002/export/pdf \
  -H "Content-Type: application/json" \
  -d '{"html":"<h1>PDF 中文测试</h1><p>今天天气不错。</p>","options":{"format":"A4"}}' \
  -o test.pdf
```

### 2.2 本地（Docker）

适合验证容器化行为、字体渲染、生产环境问题复现。

```bash
cd e:/job-project/collabedit-doc-converter

# 1) 下载预置 Chrome（首次执行，约 170MB）
bash scripts/fetch-vendor-chrome.sh
#   Windows PowerShell:
#   powershell -ExecutionPolicy Bypass -File scripts\fetch-vendor-chrome.ps1

# 2) 本地编译 dist（镜像里 COPY dist/ 而非 src/）
pnpm install --frozen-lockfile
pnpm build

# 3) 构建镜像 + 启动
docker compose build converter
docker compose up -d

# 4) 验证
docker compose logs -f converter
curl http://localhost:3002/health
```

## 3. 生产环境（联网部署）

生产机有外网（能访问 npmjs / debian mirror / CDN）时的标准流程。

### 3.1 准备

```bash
# 仅首次或 Chrome 版本升级时：下载预置 Chrome
bash scripts/fetch-vendor-chrome.sh
```

### 3.2 构建 dist + 镜像

```bash
pnpm install --frozen-lockfile
pnpm build
docker compose build converter
```

### 3.3 准备 `.env`

```bash
cp .env.example .env
```

**生产 `.env` 关键设置**：

```dotenv
PORT=3002
MAX_FILE_SIZE=200mb
MAX_CONCURRENT=5
LOG_LEVEL=info
# 生产容器内 CHROMIUM_EXECUTABLE_PATH 务必留空！使用镜像自带 Chrome
#CHROMIUM_EXECUTABLE_PATH=
PDF_RENDER_TIMEOUT=60000
```

### 3.4 启动 / 重启 / 停止

```bash
docker compose up -d          # 启动（后台）
docker compose restart        # 热重启
docker compose down           # 停止并删除容器（保留镜像）
docker compose logs -f        # 查看日志
```

### 3.5 健康检查

```bash
curl http://<生产机 IP>:3002/health
# {"status":"ok","chromium":false,"unoserver":false,"timestamp":"..."}
```

> `chromium:false` 在**首次请求前**是正常的，表示 Puppeteer 尚未懒加载启动 Chrome。处理一次 PDF 导出后会变为 `true`。

## 4. 生产环境（离线部署，无网或弱网）

当**生产机无法访问外网**（或网络极不稳定）时使用。思路：

1. 在**打包机**（有网）build 好镜像 → 导出为 tar 文件
2. 把 tar + 配套文件复制到**离线机**
3. **离线机** `docker load` 导入 → `docker compose up`

### 4.1 打包机：构建并导出镜像 tar

**前置**：

- 打包机有外网
- 安装了 Docker / docker compose、Node.js、pnpm

**一键脚本**：

```bash
cd e:/job-project/collabedit-doc-converter
bash scripts/offline-deploy.sh save
```

该脚本会依次执行：

1. `bash scripts/fetch-vendor-chrome.sh` —— 下载 Chrome zip 到 `vendor/`
2. `pnpm install --frozen-lockfile && pnpm build` —— 生成 `dist/`
3. `docker pull node:20-bookworm-slim && docker compose build converter`
4. `docker save node:20-bookworm-slim collabedit-doc-converter-converter:latest -o doc-converter-images.tar`

完成后得到 `doc-converter-images.tar`（~820MB；`docker save` 输出的 tar 内每层为 gzipped layer，约为展开后镜像体积 2.5GB 的 1/3）。

**手动分步版**（如脚本不方便使用）：

```bash
# 1) 下载 Chrome
bash scripts/fetch-vendor-chrome.sh

# 2) 本地编译
pnpm install --frozen-lockfile
pnpm build

# 3) 拉基础镜像（减少首次 build 时间）
docker pull node:20-bookworm-slim

# 4) 构建镜像
docker compose build converter

# 5) 导出 tar（包含基础镜像 + 应用镜像）
docker save \
  node:20-bookworm-slim \
  collabedit-doc-converter-converter:latest \
  -o doc-converter-images.tar
```

### 4.2 离线生产机：导入镜像并启动

**需要从打包机复制的文件**：

```
doc-converter-images.tar     # 镜像 tar（~820MB，载入后展开为 ~2.5GB 镜像）
docker-compose.yml
.env                         # 按生产需求调整后的环境变量
fonts/                       # 可选：如果希望用 volume 挂载热更新字体
```

> 字体已打进镜像（Dockerfile 里 `COPY fonts/`），通常**无需**再单独复制 `fonts/`。
> 仅当希望后续增量更新字体而不重建镜像时才挂载 `./fonts` 到 `/usr/share/fonts/custom`。

**启动步骤**：

```bash
cd /path/to/collabedit-doc-converter

# 一键脚本：
bash scripts/offline-deploy.sh load
```

**或手动分步**：

```bash
# 1) 导入镜像
docker load -i doc-converter-images.tar

# 2) 验证镜像已在本地
docker images | grep converter

# 3) 启动服务
docker compose up -d

# 4) 验证
docker compose ps
docker compose logs -f converter
curl http://localhost:3002/health
```

### 4.3 升级发布（替换镜像）

以后升级只需要：

1. 在**打包机**重新跑 `bash scripts/offline-deploy.sh save`（会覆盖 `doc-converter-images.tar`）
2. 把新 tar 传到离线机
3. 离线机执行：

```bash
docker compose down                              # 停掉旧容器
docker load -i doc-converter-images.tar          # 导入新镜像（同 tag 会覆盖）
docker compose up -d                             # 用新镜像启动
docker image prune -f                            # 清理被替换的旧镜像（可选）
```

## 5. 环境变量说明

在 `.env`（或 `docker-compose.yml` 的 `environment`）中配置：

| 变量 | 默认值 | 说明 |
|---|---|---|
| `PORT` | `3002` | HTTP 监听端口 |
| `MAX_FILE_SIZE` | `200mb` | Express body 最大大小（DOCX 单文件上限） |
| `MAX_CONCURRENT` | `5` | 并发 import/export 任务数（超出进队列） |
| `IMPORT_TIMEOUT` | `30000` | 单次 DOCX → DocModel 超时（ms） |
| `EXPORT_TIMEOUT` | `120000` | 单次导出超时（ms） |
| `LOG_LEVEL` | `info` | 日志级别 |
| `CHROMIUM_EXECUTABLE_PATH` | 空 | Chrome 可执行文件路径。容器内留空；本地开发可指向宿主机已有 Chrome |
| `PDF_RENDER_TIMEOUT` | `60000` | 单次 PDF 渲染超时（ms） |

> Docker 场景下 `docker-compose.yml` 已强制 `CHROMIUM_EXECUTABLE_PATH=` 空，防止宿主机 `.env` 的 Windows 路径被透传进容器导致 `launch` 失败。

## 6. 常用运维命令

```bash
# 查看容器
docker compose ps

# 进入容器排查
docker compose exec converter sh

# 容器内验证 Chrome 可执行
/home/node/.cache/puppeteer/chrome/linux-147.0.7727.57/chrome-linux64/chrome --version

# 实时日志
docker compose logs -f converter

# 只看最近 100 行
docker compose logs --tail=100 converter

# 重启（不重建）
docker compose restart converter

# 重建镜像 + 重启
docker compose build converter && docker compose up -d

# 强制不用缓存重建（慢，调试用）
docker compose build --no-cache converter

# 清理失败 build 的中间层
docker builder prune -f
```

## 7. 常见问题 FAQ

### Q1：`pnpm install` 阶段卡在 Puppeteer 下载 Chrome

**现象**：

```
Downloading binary from https://storage.googleapis.com/chrome-for-testing-public/...
```

国内网络访问 Google 源会超时或被切。

**解决**：

```bash
# 方案 A：跳过下载，使用宿主机 Chrome
PUPPETEER_SKIP_DOWNLOAD=true pnpm install

# 方案 B：换国内镜像
PUPPETEER_DOWNLOAD_BASE_URL=https://cdn.npmmirror.com/binaries/chrome-for-testing pnpm install
```

---

### Q2：Docker build 时 `chrome --version` 报错缺 `libXfixes.so.3` 或其他 `.so`

**现象**：

```
error while loading shared libraries: libXfixes.so.3: cannot open shared object file
```

Chrome 运行时需要系统动态库，Debian slim 默认没装全。

**解决**：在 `Dockerfile` 层 1 的 `PKGS` 列表里加上相应包。若缺其他库可用：

```bash
docker compose exec converter ldd /home/node/.cache/puppeteer/chrome/linux-147.0.7727.57/chrome-linux64/chrome \
  | grep "not found"
```

定位缺失 `.so`，再用 `apt-file search libXxx.so` 反查包名。

---

### Q3：Docker build 时下载 Chrome 中途 `SocketError: other side closed`

**现象**：`pnpm install` 到 puppeteer install 脚本阶段，~10MB 处连接被切。

**根因**：容器访问 `cdn.npmmirror.com` 大文件不稳定。

**解决**：本项目已采用"宿主机预下载 + COPY"方案（Dockerfile + `scripts/fetch-vendor-chrome.*`），不受此问题影响。如仍遇到，检查 `vendor/chrome-linux64.zip` 是否存在且 ~170MB：

```bash
ls -lh vendor/chrome-linux64.zip
# 若缺失或损坏，重新跑：
bash scripts/fetch-vendor-chrome.sh
```

---

### Q4：corepack 拉 pnpm 时失败

**现象**：`corepack prepare pnpm@latest --activate` 失败。

**解决**：Dockerfile 已内置 5 次 retry + 指向 npmmirror。如仍失败，可手动改为 `RUN npm install -g pnpm@10`（使用已配置的 npmmirror registry）。

---

### Q5：Docker build 时 apt-get install 频繁 EOF 或 502

**现象**：

```
500 reading HTTP response body: unexpected EOF
502 Bad Gateway
```

**解决**：

- Dockerfile 已经启用 `Acquire::Retries "5"` 并套了 5 次 for 循环 retry
- `docker-compose.yml` build 用了 `network: host`
- 经验上 deb.debian.org 官方源 + CDN 在 `network: host` 下最稳

---

### Q6：PDF 里中文变成方框 / 豆腐块

**原因**：字体未加载。

**排查**：

```bash
docker compose exec converter fc-list | grep -i -E "sim|wqy|noto" | head
```

应能看到 `simsun.ttc`、`wqy-zenhei` 等。若没有：

- 确认 `./fonts/` 目录里有相应 ttf/ttc 文件
- 重建镜像或确认 `docker-compose.yml` 的 volume 挂载生效
- HTML 中 `font-family` 写对名字（如 `simsun`、`宋体`）

---

### Q7：Chrome 版本升级（puppeteer 升级后）

升级 `puppeteer` 时通常会绑定新的 Chrome for Testing 版本。Puppeteer 启动时会报：

```
Could not find Chrome (ver. 148.0.xxxx.yy)
```

**操作**：

1. 更新 `scripts/fetch-vendor-chrome.sh` 和 `fetch-vendor-chrome.ps1` 里的默认版本号
2. 更新 `Dockerfile` 的 `ARG CHROME_VERSION=新版本号`
3. 删除 `vendor/chrome-linux64.zip`
4. 重新跑 `bash scripts/fetch-vendor-chrome.sh`
5. 重建镜像 `docker compose build converter`

> 查询 puppeteer 期望的 Chrome 版本：
>
> ```bash
> node -e "console.log(require('puppeteer').PUPPETEER_REVISIONS)"
> ```

---

### Q8：健康检查返回 `chromium:false`，是不是坏了？

**不是**。Puppeteer 采用懒加载，在第一次 `/export/pdf` 请求前 Chrome 没启动，所以 `/health` 里 `chromium` 字段为 `false`。一旦处理过一次导出请求，该字段就会变 `true`（直到 Chrome 崩溃被销毁）。

---

### Q9：如何彻底清理并重来

```bash
docker compose down
docker image rm collabedit-doc-converter-converter:latest
docker builder prune -a -f               # 清 BuildKit 缓存
rm -rf node_modules dist
rm -rf vendor/*.zip                      # 强制重新下载 Chrome
pnpm install --frozen-lockfile
pnpm build
bash scripts/fetch-vendor-chrome.sh
docker compose build converter
docker compose up -d
```

## 附录：文件清单

| 文件 | 作用 |
|---|---|
| `Dockerfile` | 镜像构建：系统库 + 字体 + Chrome（预置）+ node_modules + dist |

| `dist/` | 编译后 JS，`pnpm build` 产物，build 时 COPY 进镜像 |
